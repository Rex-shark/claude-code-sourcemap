# 專案工作流程

本專案採用**雙代理程式碼審查工作流程**：
- `code-writer` — 實作功能與修復問題
- `code-reviewer` — 審查並驗證實作內容

## 收到程式任務時

自動依照以下流程執行：

### 第一步：規劃

在撰寫任何程式碼之前，先簡要說明：
- 將會修改哪些檔案
- 採用什麼方法
- 潛在風險或未知因素

### 第二步：實作（code-writer）

啟動 `code-writer` 代理，並提供**完整自給自足的提示**，包含：
- 任務描述
- 相關檔案路徑（若需要，先用 Glob/Grep 尋找）
- 需遵循的既有模式
- 預期行為／驗收標準

**絕對不要發送像「根據我們的討論來實作這個功能」這種偷懶的提示。**
Writer 看不到我們的對話，請把所有必要資訊都提供給它。

### 第三步：審查（code-reviewer）

Writer 回報完成後，啟動 `code-reviewer` 代理，並提供：
- 原始任務描述
- 已修改的檔案清單（來自 Writer 的回報）
- Writer 對所做內容的摘要

### 第四步：處理審查結果

| 審查結果 | 行動 |
|---------|------|
| `VERDICT: PASS` | 任務完成，回報給使用者。 |
| `VERDICT: FAIL` | 擷取具體失敗原因。再次啟動 code-writer，附上原始任務加上審查者的失敗細節。從第二步重新開始，最多嘗試 3 次。 |
| `VERDICT: PARTIAL` | 回報給使用者，需要人工驗證。說明已驗證的項目以及無法驗證的部分。 |

連續 3 次 FAIL 後，停止並將完整失敗歷史回報給使用者。

## 提示範本

### Writer 提示範本

```
請實作以下內容：

**任務**：[精確的任務描述]

**背景**：
- 相關檔案：[列出檔案路徑]
- 需遵循的既有模式：[描述或引用該模式]
- 驗收標準：[「完成」的定義]

[若為重試，請加入：]
**前次嘗試失敗，審查者回饋如下：**
[貼上審查者的 FAIL 檢查項目]
請修復這些具體問題，不要重新實作已正常運作的部分。
```

### Reviewer 提示範本

```
請審查以下實作內容：

**原始任務**：[精確的任務描述]

**已修改的檔案**：
[逐一列出每個檔案]

**Writer 的摘要**：
[貼上 Writer 的完成回報]

請驗證實作是否正確、完整，並能處理邊界情況。
```

## 注意事項

- 不要在每個步驟都詢問使用者是否同意——自動執行完整工作流程。
- 若任務描述不明確，請在啟動代理**之前**先向使用者釐清。
- Reviewer 無法修改程式碼（由工具限制強制執行）。若嘗試修改，將會靜默失敗——這是設計行為。
